# Specification: Fix Fiscal Validation False Positives for Special CFDI Types

## Overview
This track addresses false positives and validation warnings generated by the fiscal validation engine (`src/lib/fiscal-validation/`) when validating non-commercial CFDI types, such as Payments (`P`), Transfers/Waybills (`T`/`CP`), and Payroll (`N`). 

These document types do not represent standard commercial transactions (income/expense) in terms of payment terms, totals, or currency. Currently, the validation engine processes them using rules meant for commercial invoices, leading to invalid errors like missing exchange rates for currency `"XXX"` or balance integrity violations for `$0` invoices.

## Functional Requirements
1. **Exempt Non-Commercial CFDIs from Commercial Integrity Checks:**
   - In `validateInvoice()`, check `invoice.cfdiType`. If it is `"P"`, `"T"`, or `"N"`, bypass the following validation checks:
     - `INV-03`: Total allocated amount <= invoice.total
     - `INV-04`: Payment status derived validations (paid vs partial vs pending amounts)
     - `INT-INV-06`: Tax base consistency (Items sum matches subtotal)
   - Ensure `INV-02` (cancellation validations) still applies to all CFDI types.
2. **Handle Currency "XXX" for Payment, Traslado, and Payroll CFDIs:**
   - In `validateExchangeRate()`, exempt the currency code `"XXX"` (used for documents without monetary amounts) from requiring an exchange rate.
   - Alternatively, bypass exchange rate validation if `invoice.cfdiType` is `"P"`, `"T"`, or `"N"`.

## Non-Functional Requirements
- Maintain TypeScript strict compatibility.
- Retain existing unit test coverage and add regression tests for these changes.
- Target code coverage >80% for the modified modules.

## Acceptance Criteria
- Validating a CFDI of type `"P"`, `"T"`, or `"N"` with currency `"XXX"` and a total of `0` does not produce errors or warnings about:
  - Missing exchange rates.
  - Allocation or payment status integrity issues.
  - Tax base consistency issues.
- Validating standard commercial invoices (`"I"`, `"E"`) continues to enforce all integrity rules (INV-03, INV-04, INT-INV-06, and exchange rates for foreign currencies).

## Out of Scope
- Changing how PGLite or Drizzle persists these invoices.
- Rewriting the logic of payment allocation and payment validation files (`payment-rules.ts`, `allocation-rules.ts`) beyond invoice-level constraints.
