# Track Specification: Legal-Safe Adjustments for RESICO Tax Assistance

## Overview
This track implements a series of UI, copy, and behavioral changes to ensure the platform is clearly positioned as an informational tax assistance tool. The goal is to prevent users from perceiving the system as a final authority on fiscal compliance or a direct filing system, while maintaining the data integrity features that make the tool useful.

## Functional Requirements

### 1. Terminology Refactor (UI & Metadata)
- Rename "Tax Declarations" to "Tax Estimations" or "Draft Calculations" across the entire application (navigation, headings, buttons).
- Update Audit Log metadata and descriptions to use advisory language (e.g., "Updated calculation preferences" instead of "Updated tax compliance rules").
- Ensure all references to "Compliance" or "Fiscal Validity" are replaced with "Data Consistency" or "RESICO Best Practices".

### 2. Dashboard & Reporting Disclaimers
- Implement a persistent disclaimer in the "Tax Estimations" (formerly Declarations) section stating that values are for informational purposes only.
- Add footer disclaimers to any summary views emphasizing that the application does not submit data to the SAT or guarantee fiscal correctness.

### 3. Validation Logic Reframing
- Maintain existing strict validation rules (e.g., Cash-Basis invariants) via the `FiscalValidator` to ensure data quality.
- Update all error messages generated by `FiscalValidator` to frame issues as "Data Consistency Warnings" or "Suggested Corrections" rather than "Compliance Violations".
- Ensure that blocking state transitions (preventing save) clearly explains that the block is for "Data Integrity" within the app's calculation logic.

### 4. Metadata & Internal Documentation
- Update internal constants, types, and comments where "fiscal" or "compliance" terminology is used to reflect the "assistance/estimation" nature.

## Non-Functional Requirements
- **Consistency**: The new terminology must be applied globally to avoid conflicting messages to the user.
- **No Data Loss**: Changes to validation messaging must not relax the underlying data integrity constraints.

## Acceptance Criteria
- [ ] "Tax Declarations" is no longer visible in the UI, replaced by "Tax Estimations".
- [ ] Error messages for RESICO rule violations use advisory language.
- [ ] Audit logs reflect the renamed terminology.
- [ ] Clear disclaimers are visible in calculation summary views.

## Out of Scope
- Changing the underlying business logic of the `FiscalValidator` (only the messaging/framing is changed).
- Changing database schema column names (unless necessary for critical UI reflection).
